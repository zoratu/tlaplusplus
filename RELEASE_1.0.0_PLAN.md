### Bugs uncovered by T2 (block 1.0.0; fix before next phase)

- [x] **T2.1. SOUNDNESS: compiled-expr parser fails on nested `[r EXCEPT !.f = v]`.** Closed — `try_parse_except` now scans for `EXCEPT` only at top-level bracket depth via the new `find_top_level_except` helper. `[[r EXCEPT !.a = 0] EXCEPT !.b = 0]` now compiles to `FuncExcept(FuncExcept(r, [a→0]), [b→0])` and evaluates to `[a |-> 0, b |-> 0]`. See log entry `T2.1 + T2.2 — nested / record-field EXCEPT`.
- [x] **T2.2. SOUNDNESS: compiled-expr parser fails on EXCEPT inside a record-literal field value.** Closed by the same fix as T2.1 — the inner EXCEPT inside `[a |-> 0, b |-> ([r EXCEPT !.a = 0]).a]` is no longer matched by `try_parse_except`'s outer scan, so the record literal path correctly takes over. Evaluates to `[a |-> 0, b |-> 0]` matching the interpreter. See log entry `T2.1 + T2.2`.
- [x] **T2.3. SOUNDNESS: interpreter type-errors on `<range> \subseteq <set>` (and `\intersect`).** Closed — `..` (TLA+ precedence 9-9) was lumped into the relational/comparison tier in the interpreter (`split_top_level_comparison`) and placed *above* the set-op block in the compiler cascade. Both directions silently misparsed `n..m \subseteq S` and `S \union n..m` shapes. Fix: removed `..` from comparison patterns and added a dedicated `split_top_level_range` helper in the interpreter; reordered the compiler cascade so set ops (8-8) come before `..` (9-9) come before additive (10-10). 8 regression tests, 13/13 diff harness pass, proptest harness now green across 9 seeds. See log entry `T2.3 — \`..\` precedence vs set ops`.
- [x] **T2.4. SOUNDNESS: compiled-vs-interpreted divergence on `<<(-1 - (r).a), 0>>` and similar arithmetic-with-unary-minus inside tuple/sequence syntax.** Closed — `compile_expr`'s `-` arm called `split_binary_op(expr, "-")` which returns the FIRST top-level `-`. For `-1 - (r).a` that's the leading unary minus (left = ""), the existing `left_ends_with_value` guard rejected the split, and the caller did not keep scanning, so the binary `-` was lost; the expression then fell through to `find_record_access_dot` and was mis-compiled as `RecordAccess(Unparsed("-1 - (r)"), "a")`. Same family for `-((LET ... IN ...))`, which had no rule to wrap unary minus on a parenthesized non-literal subexpression and ended up as `Unparsed`. Fix: new `find_binary_minus_split` helper that walks past unary `-` positions, plus a unary-minus arm that wraps `-X` as `Neg(compile_expr(X))` after binary subtraction has been ruled out. 5 regression assertions in `t2_4_unary_minus_does_not_swallow_binary_minus`, 5 independent proptest runs at `PROPTEST_CASES=2048` all green (uniform + swarm), 13/13 diff harness pass, 727 tests pass. See log entry `T2.4 — unary minus swallows binary subtraction`.

## Phase 2 — High-leverage performance

- [x] **T5. Symbolic Init enumeration.** Z3-backed enumeration for filtered record sets that currently brute-force. Behind `symbolic-init` feature flag (default off; requires libz3-dev + clang). Module: `src/tla/symbolic_init.rs`. Wired into `eval_set_expression` immediately before brute-force fallback. Translator handles `/\`, `\/`, `~`, `=>`, `<=>`, `=`, `#`, `<`, `<=`, `>=`, `>`, `+`, `-`, `*`, `\in {literal,...}`, `\in lo..hi`, field access `tup.f`, and quantifiers expanded over finite literal domains. Constant folding via the outer evaluator handles arbitrary subexpressions that don't reference the bound record variable. Domain identifier resolution: when the record-set domain is a name like `Can` rather than a literal `[f1: D1, ...]`, the new code resolves it through `ctx.definitions` so the fast path also fires. Benchmark on `TightCan` (5-field record set with sortedness + sum constraint): N=15 brute-force 14.71s → symbolic 1.40s (10.5x); N=20 brute-force 58.42s → symbolic 1.41s (41x); N=40 brute-force fails on eval budget → symbolic 1.6s. Correctness gate (proptest equivalence + brute-force-on-small) green at 64 random cases. 671 tests pass with the feature on (vs 663 without; +8 = 6 unit + 2 integration). Einstein still falls back gracefully (its Init shape is `var \in {p \in PermutationSet : pred(p)}` over sequences — outside the supported subset). See `### T5` log entry for details and follow-up T5.N entries.

- [x] **T5.5. SHIPPED v1.1.0.** Joint Init+Solution symbolic encoding. For Einstein specifically, the cleanest route to "complete in seconds" is to encode all 5 sequence variables, all 5 distinctness constraints, AND all 15 Solution rules as a single Z3 query, then solve once.
