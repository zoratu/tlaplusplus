# T10.2 — Streaming SCC discovery for liveness on 100M+-state graphs

**Status**: Phase 1 (oracle + opt-in flag) shipped in v1.1.0 development; Phase 2 (in-exploration streaming) deferred. This document covers what shipped and the concrete plan for the rest.

## 1. Why we need this

T10 cut liveness post-processing from 64 s to 115 ms on the 32K-state `LivenessBench` by replacing state-keyed Tarjan + naive fairness scan with a fingerprint-keyed iterative pipeline. That fix solved the *post-processing* time complexity, but it left in place the structural assumption that **the entire transition graph fits in memory**. For 100M+-state distributed-protocol verifications (the size that motivates `tlaplusplus` v1.x — see the README's c8g.metal-48xl scaling claims), that assumption breaks. A back-of-envelope:

- 100M states × ~5 successors/state = 500M `LabeledTransition<S>` records
- Each record holds two `S` clones plus an action `String` — typical TLA+ state is 200-500 bytes serialized; in-memory representation 1-2 KB
- 500M × ~2.5 KB ≈ **1.25 TB** of transition map alone

Even with disk spilling we'd be writing terabytes of transition data only to consume it once during liveness post-processing. The textbook fix is **on-the-fly liveness checking**: weave the cycle detection into the worker exploration so the transition map is never materialized.

## 2. Algorithm choice — nested DFS (Courcoubetis-Vardi-Wolper-Yannakakis 1992)

Three viable streaming algorithms:

| Algorithm | Memory per state | Parallel? | Notes |
|---|---|---|---|
| Nested DFS (CVWY '92) | 2 bits (color) | Multi-DFS variants exist (Holzmann '11) | Industry standard (SPIN, DiVinE) |
| One-Way Catch-Them-Young (Schwoon-Esparza '05) | 2 bits + accept-set | Better worst-case for accepting cycles | More complex |
| Streaming Tarjan with bounded stack | O(SCC width) | Hard to parallelize correctly | Doesn't compose with work-stealing |

We pick **nested DFS** because:
1. The 2-bit color tag is the absolute minimum-memory representation.
2. The algorithm is robust to multi-DFS parallelization (each worker runs an independent blue DFS; reds can be shared via a shared red color set).
3. SPIN's track record over 30 years.

The iterative reference implementation lives in `src/streaming_scc.rs` and is correctness-tested on 8 synthetic graph shapes (cycle-with-accept, cycle-without-accept, accepting-DAG, multi-component, self-loop, deep-chain stack-safety, FP-adjacency-adapter, dead-end-vs-cycle disambiguation).

## 3. Phase 1 — what shipped (oracle mode)

`--liveness-streaming` is wired into the runtime. When enabled, after the existing Tarjan-based fairness check completes, the runtime:

1. Walks all SCCs the Tarjan path produced.
2. Marks every fingerprint in any "violating" SCC as `accepting`.
3. Runs nested-DFS over the same fingerprint adjacency map, using the marked set as the acceptance predicate.
4. Cross-validates: if `oracle_found_cycle != tarjan_found_cycle`, prints a `[T10.2 oracle] DIVERGENCE` diagnostic.

This is the staging ground. The **goal of phase 1 is correctness validation**: prove the nested-DFS oracle agrees with Tarjan on every spec the corpus exercises before we lift it into the worker hot loop. Memory usage is unchanged in phase 1 — the oracle still consumes the existing adjacency map.

Files touched in phase 1:
- `src/streaming_scc.rs` (new): `LivenessGraph` trait, iterative `nested_dfs()`, `FingerprintAdjacencyGraph` adapter, 8 unit tests.
- `src/runtime.rs`: `EngineConfig.liveness_streaming` field, oracle block at the end of liveness post-processing.
- `src/main.rs`: `--liveness-streaming` CLI flag.
- `src/lib.rs`: module export.

## 4. Phase 2 — in-exploration streaming (the actual scaling fix)

This is the real work. The phase-1 oracle only validates; it doesn't reduce memory. To kill the transition map we have to **fold the DFS into the worker loop**.

### 4.1 Data structure changes

Drop:
- `labeled_transitions: Arc<DashMap<u64, Vec<LabeledTransition<S>>>>` (current, dies in phase 2).

Add:
- `colors: Arc<PageAlignedColorMap>` — a 2-bit color per fingerprint, page-aligned, NUMA-shard-placed (mirror the `PageAlignedFingerprintStore` design). 100M states × 2 bits = 25 MB total, NUMA-distributed.
- Per-worker `blue_stack: Vec<u64>` — fingerprints on the current DFS path. Size bounded by graph diameter (typically ≤1M for distributed protocols).
- Per-worker `accept_witness: Option<(u64, ActionLabel)>` — set when a fairness obligation has been pending too long (parameterized per fairness constraint).
- A shared `red_set: Arc<PageAlignedColorMap>` — separate 2-bit map, written by red DFS (CAS for concurrency).

Acceptance predicate (from a fairness constraint): a state `s` is accepting under `WF_vars(A)` iff there is a path ending at `s` that has not taken `A` for ≥ N steps where `N` ≥ graph diameter. Implementation: piggyback on the worker's blue stack — track per-action "last fired at depth" and flag accepting when depth − last_fired exceeds threshold. (A more rigorous treatment uses a Buchi-product construction with a 1-bit accepting state; for our weak/strong fairness shapes the threshold heuristic is equivalent and avoids the product blowup.)

### 4.2 Worker-loop changes

Replace the BFS hot loop in `src/runtime.rs` (around line 2440-2510) with a DFS loop. Pseudocode:

```rust
// Per-worker loop (replaces current BFS body)
loop {
    let frame = match worker_dfs_stack.pop() {
        Some(f) => f,
        None => match steal_or_terminate() { ... }
    };
    let (state, succs, idx) = frame;

    if idx == 0 {
        // First visit: enter (color = Cyan), push on blue stack
        if !colors.cas(fp(state), White, Cyan) { continue; } // already visited
        blue_stack.push(fp(state));
        check_invariants(state)?;
        update_fairness_witness(state, &mut accept_witness);
    }

    // Visit next successor or post-process
    if idx < succs.len() {
        let next = succs[idx];
        worker_dfs_stack.push((state, succs, idx + 1));
        worker_dfs_stack.push((next, model.next_states(next), 0));
    } else {
        // Post-order: maybe launch red DFS, then mark Blue and pop blue_stack
        if accept_witness.is_some() {
            if let Some(cycle) = red_dfs_from(fp(state), blue_stack, &colors, &red_set) {
                report_violation(cycle);
                return;
            }
        }
        colors.cas(fp(state), Cyan, Blue);
        blue_stack.pop();
    }
}
```

### 4.3 Trace reconstruction

The current path collects parent fingerprints into a `parent_map: DashMap<u64, u64>` to rebuild traces from any reachable state. With DFS the trace is **already on the blue stack**: when we report a violation, the lasso is `(blue_stack[..witness_idx], blue_stack[witness_idx..], cycle_via_red_trail)`. This eliminates parent-map memory entirely (currently bounded by `--max-trace-states` = 10M default).

### 4.4 Distributed work-stealing

The hard problem. Today's work-stealing distributes BFS frontier states freely between workers — order doesn't matter for correctness. With DFS, **two workers cannot share state ownership of a blue stack** without breaking the algorithm's invariants. Options:

1. **Per-worker partitioned graphs (recommended).** Hash-partition the state space by fingerprint mod N_workers. Each worker owns a partition; cross-partition edges are queued for the owner via a one-way channel. Work-stealing happens only at the partition boundary (steal a queued cross-edge, not a stack frame). This preserves DFS local invariants and requires only a small protocol on top of the existing distributed work-stealer.
2. **Speculative DFS with rollback.** Allow stealing of stack tails, but maintain a per-edge "commit" log and roll back on conflict. Complex; bug-prone.
3. **Fall back to single-worker DFS for liveness.** Acceptable if liveness verification can be the bottleneck; not acceptable for our 192-worker target.

Option 1 is the design. Implementation cost: ~1 week to reshape the work-stealer's queue API + a bring-up week against `LivenessBench` and one corpus liveness spec.

### 4.5 Termination detection

Current termination: all workers idle + global queue empty. With per-partition DFS: each worker tracks its DFS stack; termination is "all workers' DFS stacks empty + all cross-partition queues empty + all red DFS frontiers drained". Add a phased commit (akin to the bloom-exchange phase in the distributed module).

## 5. Test strategy

### 5.1 Unit tests (phase 1, shipped)
- 8 synthetic graphs in `src/streaming_scc.rs::tests`: trivial accepting cycle, cycle-without-accept, accepting-DAG (no cycle), multi-component, self-loop accept, deep-chain stack-safety (200K nodes), FP-adapter, dead-end-vs-cycle.

### 5.2 Integration tests (phase 1, validated via oracle)
- `tests/wrapper_next_fairness_t1_3.rs`: existing fixtures must produce identical verdicts under `liveness_streaming = true` (one passes fairness, one detects WF_vars(NeverFires) violation).
- New `tests/streaming_scc_oracle.rs` (phase 1.5, optional): runs the same fairness specs with both modes and asserts no `[T10.2 oracle] DIVERGENCE` lines on stderr.

### 5.3 Phase 2 tests
- Add a 1M-state synthetic spec (extend `LivenessBench` to N=20). Validate: same verdict, peak RSS < 1.5 GB (vs ~ 30 GB for the materialized-transitions path at this size).
- Property-based test: generate random labeled DAGs + planted accepting cycles; assert nested-DFS in-runtime mode finds them iff they exist.

### 5.4 Differential gate vs. Tarjan
- Cross-validate every corpus-fairness spec. Hard requirement: zero divergences before swapping the default to streaming mode.

## 6. Corpus revalidation plan

The fairness path is exercised by ~12 specs in the public corpus (Lamport's mutex variants, Paxos liveness, the WrapperNext fixtures). Plan:

1. Run the entire corpus with `--liveness-streaming` enabled (oracle mode). Confirm zero divergences.
2. Switch the corpus runner to use `--liveness-streaming` as the primary check on a feature branch. Run full corpus on a c8g.metal-24xl. Expect identical verdicts and ≤ identical wall-time (oracle mode is additive cost).
3. After the in-exploration variant lands, re-run with the BFS path disabled. Validate identical verdicts AND demonstrate ≥10x peak-RSS reduction on the largest fairness spec.

Risk factors:
- **Distributed DFS semantics.** Option 1 (partitioned ownership) has not been prototyped at scale; the cross-partition queue may need a back-pressure protocol of its own.
- **Fairness threshold tuning.** The depth-thresholded acceptance is a heuristic; some pathological specs may need true Buchi-product construction. Add a fallback path that falls through to the existing Tarjan-based check when the heuristic flags but cannot prove.
- **Trace reconstruction across partitions.** When the cycle straddles partitions, the lasso prefix may live on one worker's blue stack and the cycle body on another's. Need a phase-3 "trace reconciliation" step that collects fragments from all participating workers.
