# T10.2 phase 2 — refined design (in-exploration streaming nested DFS)

**Status**: Phase 1 (oracle + opt-in `--liveness-streaming`) shipped in v1.1.0. Phase 2 (in-exploration streaming, the actual memory win) **not yet implemented**. This document refines `docs/T10.2-streaming-scc-design.md` with the concrete data structures, distributed protocol message types, exact runtime integration points, and corpus revalidation plan that the original doc left at the "TODO" level.

The goal is to give the next implementer enough specificity that the work fits in the original 5-week estimate without spending the first week re-discovering the integration surface.

## 0. Why this refinement was written

The original design doc estimated the work at ~5 calendar weeks for a single engineer and called out three risk factors (distributed DFS semantics, fairness threshold tuning, cross-partition trace reconstruction). On a closer read of the v1.1.0 codebase those risk factors are concrete enough to pin down before any code is written. In particular:

- The `Model` trait already exposes `fairness_constraints() -> Vec<FairnessConstraint>` and `next_action_name() -> Option<&str>` (`src/model.rs:80,94`). The "fairness threshold semantics" risk reduces to a small adapter that maps each `FairnessConstraint` to an acceptance predicate over `(state, depth, last_fired)`, not to an open research question.
- The distributed protocol (`src/distributed/protocol.rs`) is a single `Message` enum with a `bincode` round-trip test per variant. Adding the partition queue is purely additive (new variants + handler arms), not a protocol redesign.
- The runtime's BFS hot loop is wedged between `worker_labeled_transitions` (`src/runtime.rs:2586`) and the `process_batch` closure (`:2651`). The DFS rewrite is bounded to the worker body in `run_model`; the surrounding cgroup/NUMA/auto-tune/checkpoint scaffolding is unaffected.

What the original doc did not address, and what this refinement adds:

1. The **exact byte layout** and atomic-CAS shape for `PageAlignedColorMap` (mirroring `PageAlignedFingerprintStore` in `src/storage/page_aligned_fingerprint_store.rs:1272`).
2. The **exact `Message` enum additions** for the cross-partition queue and the wire-format checklist.
3. The **integration points in `src/runtime.rs`** with line numbers for the v1.1.0 commit.
4. A **revalidation plan** that names the existing test fixtures and the concrete spec to use for the >50% RSS benchmark.
5. A **fallback escape hatch** so the change can ship safety-only first, with fairness gated behind a separate flag.

## 1. `PageAlignedColorMap` — exact data structure

### 1.1 Memory layout

Mirror `FingerprintShard` in `src/storage/page_aligned_fingerprint_store.rs`. Differences:

| Aspect | `FingerprintShard` | `ColorShard` (new) |
|---|---|---|
| Slot type | `AtomicU64` (full fingerprint) | `AtomicU64` packed: 32 × 2-bit color slots |
| Slot count | `capacity` fingerprints | `capacity` fingerprints, addressed at sub-word granularity |
| Page alignment | 2 MB hugepage mmap | 2 MB hugepage mmap (same) |
| NUMA placement | `mbind(MPOL_PREFERRED, node)` | `mbind(MPOL_PREFERRED, node)` (same) |
| Resize | Seqlock + atomic pointer swap (line ~600) | **No resize.** Color map is sized once at startup from `--fp-expected-items`. |

Why no resize: the color map is sized to track the same fingerprint domain as the fingerprint store, which already resizes. If the fingerprint store grows past the color map's capacity, the color map allocates a parallel "overflow" shard at next worker quiescence point. The expected steady-state path never hits this — at `--fp-expected-items 100M` the color map is 25 MB; doubling it costs 25 MB more, not 1.25 TB.

### 1.2 The 2-bit encoding

```rust
#[repr(u8)]
enum Color {
    White       = 0b00,  // never visited
    Cyan        = 0b01,  // on current blue DFS path (cycle witness target)
    Blue        = 0b10,  // blue DFS finished
    Red         = 0b11,  // visited by red DFS
}
```

Slot addressing within a `u64`:
```
fp -> shard_idx = fp >> 32                 // shard selection (mirror existing)
fp -> within_shard = (fp as u32) % capacity
fp -> word_idx = within_shard / 32
fp -> bit_offset = (within_shard % 32) * 2
```

CAS path:
```rust
fn cas_color(&self, fp: u64, expected: Color, new: Color) -> Result<(), Color> {
    let (shard_idx, word_idx, bit_offset) = self.locate(fp);
    let word = &self.shards[shard_idx].words[word_idx];
    let mask = 0b11u64 << bit_offset;
    loop {
        let cur = word.load(Ordering::Acquire);
        let cur_color = Color::from_bits((cur >> bit_offset) & 0b11);
        if cur_color != expected { return Err(cur_color); }
        let next_word = (cur & !mask) | ((new as u64) << bit_offset);
        match word.compare_exchange_weak(cur, next_word, Ordering::AcqRel, Ordering::Acquire) {
            Ok(_) => return Ok(()),
            Err(_) => continue,  // retry; another worker won the CAS
        }
    }
}
```

## 2. Worker DFS loop — exact integration in `src/runtime.rs`

### 2.1 The replacement region

Today's BFS body in `src/runtime.rs` runs from approximately line 2440 (after the worker pops a state) through line 2810 (where successors are batched into `process_batch` and re-pushed to the local deque). The replacement is constrained to that region; everything outside (cgroup setup, NUMA pinning, auto-tuner, checkpoint coordination, distributed-stealer integration, signal handling, progress reporting) stays as-is.

### 2.2 Per-worker DFS state — the new struct

```rust
struct WorkerDfsState<S> {
    /// Explicit DFS frame stack. Each frame is (state, successors, next_idx).
    /// State clones are unavoidable here because DFS post-order needs to
    /// re-visit `state` after its successors return.
    stack: Vec<DfsFrame<S>>,
    /// Fingerprints on the current DFS path, in push order. Bounded by
    /// graph diameter (typically <= 1M for distributed protocols; we cap
    /// at --max-dfs-depth and emit a "DFS truncated" warning if exceeded).
    blue_path_fps: Vec<u64>,
    /// Per-action "last fired at depth" map for fairness-threshold acceptance.
    /// Allocated once per worker (one entry per action in the model).
    last_fired_depth: SmallVec<[Option<usize>; 8]>,
    /// Set on accept; consumed when the post-order red DFS launches.
    accept_pending: bool,
}

struct DfsFrame<S> {
    state: S,
    successors: Vec<S>,
    next_idx: usize,
    /// Action label that produced this frame (for trace reconstruction).
    via_action: Option<u32>,  // index into model's action table
}
```

Memory budget per worker: `stack` peaks at graph diameter × (state size + Vec<S> capacity). On a 1M-state liveness spec with diameter ~10K and 200-byte states, that's ~2 MB per worker. At 192 workers, ~400 MB total — orders of magnitude smaller than today's `labeled_transitions: DashMap<u64, Vec<LabeledTransition<S>>>` (which on the same model is ~5 GB before disk spill).

### 2.3 The replacement body (pseudocode, with named helpers)

```rust
'worker_loop: loop {
    if dfs_state.stack.is_empty() {
        // Stack drained -> try to take a new root from the cross-partition
        // inbox or steal a root from a peer worker. Same termination logic
        // as the BFS path (see section 2.4).
        match acquire_dfs_root(&dfs_state, &cross_partition_inbox, &steal_targets) {
            AcquireResult::Root(s)     => enter_blue(&mut dfs_state, s, fp_store, color_map),
            AcquireResult::Terminate   => break 'worker_loop,
            AcquireResult::Backoff(d)  => { std::thread::sleep(d); continue; }
        }
    }

    let frame = dfs_state.stack.last_mut().unwrap();

    // Visit next successor or perform post-order.
    if frame.next_idx < frame.successors.len() {
        let succ = frame.successors[frame.next_idx].clone();
        frame.next_idx += 1;

        // Invariant check on edge (matches BFS path semantics).
        worker_model.check_invariants(&succ)?;

        let succ_fp = worker_model.fingerprint(&succ);

        // Cross-partition routing: if succ_fp's home partition isn't ours,
        // hand it off via the partition inbox. See section 3.
        if !owns_partition(succ_fp, my_partition_id, partition_count) {
            cross_partition_outbox.push(succ_fp, &succ, my_partition_id);
            continue;
        }

        // Try to claim this state for our blue DFS. CAS White -> Cyan.
        match color_map.cas(succ_fp, Color::White, Color::Cyan) {
            Ok(()) => {
                // Fresh -> push frame.
                dfs_state.blue_path_fps.push(succ_fp);
                update_last_fired(&mut dfs_state.last_fired_depth, &frame.via_action,
                                  dfs_state.blue_path_fps.len());
                let succs = compute_successors(&worker_model, &succ);
                dfs_state.stack.push(DfsFrame { state: succ, successors: succs, next_idx: 0,
                                                 via_action: edge.action });
            }
            Err(Color::Cyan) => {
                // Back-edge to our own blue stack -> potential accepting cycle.
                if frame_acceptance_threshold_exceeded(&dfs_state) {
                    return Err(report_lasso(&dfs_state, succ_fp));
                }
            }
            Err(Color::Blue) | Err(Color::Red) => { /* already done; skip */ }
            Err(Color::White) => unreachable!(), // CAS would have succeeded
        }
    } else {
        // Post-order: maybe launch red DFS, then mark Blue and pop.
        let v_fp = *dfs_state.blue_path_fps.last().unwrap();
        if dfs_state.accept_pending {
            if let Some(witness) = red_dfs_streaming(v_fp, &dfs_state.blue_path_fps,
                                                     &color_map, &red_set,
                                                     &cross_partition_outbox)? {
                return Err(report_lasso_with_witness(&dfs_state, witness));
            }
            dfs_state.accept_pending = false;
        }
        let _ = color_map.cas(v_fp, Color::Cyan, Color::Blue);  // ok if lost to a peer
        dfs_state.blue_path_fps.pop();
        dfs_state.stack.pop();
    }
}
```

### 2.4 What stays from the BFS path

These BFS-path facilities transfer unchanged:

- `worker_queue.worker_idle(...)` and `worker_queue.has_pending_work()` — DFS uses the same APIs but now to track partition-inbox depth, not local-deque depth.
- The fingerprint store CAS for dedup (`fp_store.insert(...)`) — DFS still updates the global FP store; the color map is **additive**, not a replacement. (Tradeoff: 25 MB extra. Avoiding the FP store would require teaching the FP store to also store color, which conflates two concerns.)
- All invariant-check, deadlock-check, and constraint-filter code (the four sites under `worker_model.check_*` in `src/runtime.rs:2620-2640`).
- Checkpoint coordination (`pause.is_paused()`, `pause.worker_paused(id)`).
- The auto-tuner's `throttle.should_yield(worker_id)` check (DFS workers yield at the top of the loop, same place BFS yields).

### 2.5 What the BFS path keeps for non-fairness specs

**The DFS path is gated on `--liveness-streaming-exploration` AND `model.has_fairness_constraints()`.** Specs without fairness continue to use the existing BFS path with no change. This preserves the "default behavior unchanged on every existing spec" requirement and is the safety net for the rollout.

## 3. Distributed cross-partition queue protocol

### 3.1 Partition assignment

`partition_id(fp) = fp mod partition_count`. `partition_count = num_workers` for in-process partitioning; `partition_count = num_workers × num_nodes` for cluster-mode partitioning (each node owns a contiguous block).

### 3.2 New message variants in `src/distributed/protocol.rs`

```rust
pub enum Message {
    // ... existing variants (StealRequest, StealResponse, BloomExchange, Join,
    //     Leave, Heartbeat, Stop, TerminationToken) ...

    /// A cross-partition DFS edge. The sender encountered `state_fp` while
    /// expanding a frame; the receiver owns this fingerprint's partition
    /// and is responsible for entering blue DFS on it (or, if already
    /// Cyan/Blue/Red, recording the back-edge for cycle detection).
    PartitionEdge {
        from_node: u32,
        from_worker: u32,
        owner_node: u32,
        owner_worker: u32,
        /// zstd-compressed bincode of the state.
        state_blob: Vec<u8>,
        /// Fingerprint of state_blob (sender's home), so receiver doesn't
        /// recompute on every message.
        state_fp: u64,
        /// The action label that produced this edge, for trace reconstruction.
        via_action: u16,  // index into model action table; bincoded
        /// The sender's blue_path_fps depth when emitting this edge,
        /// used by the receiver for fairness-threshold computation.
        sender_depth: u32,
    },

    /// Acknowledgement that a PartitionEdge was processed (added to receiver's
    /// frontier OR found already-colored). Required for termination
    /// detection: the global "all queues empty" predicate has to count
    /// in-flight partition edges, mirroring the existing pending-steal
    /// counter on `DistributedStealer`.
    PartitionEdgeAck {
        from_node: u32,
        owner_node: u32,
        state_fp: u64,
        outcome: PartitionEdgeOutcome,  // ClaimedFresh | AlreadyVisited | BackEdgeToCyan
    },

    /// Cross-partition red DFS probe. Red DFS searching for a back-edge
    /// to a Cyan state may itself cross partitions; this carries the
    /// search to the partition owning `target_fp`.
    RedDfsProbe {
        from_node: u32,
        from_worker: u32,
        owner_node: u32,
        owner_worker: u32,
        seed_fp: u64,         // the accepting state that launched this red DFS
        target_fp: u64,       // the state to test for Cyan / continue red DFS from
        trail: Vec<u64>,      // fps on the red trail so far (for witness reconstruction)
    },

    /// Response to RedDfsProbe.
    RedDfsResponse {
        from_node: u32,
        seed_fp: u64,
        outcome: RedDfsOutcome,  // FoundCyanWitness(witness_fp) | NotFound | RedDfsTruncated
        trail_extension: Vec<u64>,  // fps to append to the requester's trail
    },
}

#[derive(Serialize, Deserialize, Debug, Clone, Copy)]
pub enum PartitionEdgeOutcome { ClaimedFresh, AlreadyVisited, BackEdgeToCyan }

#[derive(Serialize, Deserialize, Debug, Clone)]
pub enum RedDfsOutcome { FoundCyanWitness(u64), NotFound, RedDfsTruncated }
```

### 3.3 In-process channels (single-node case)

Use `crossbeam_channel::bounded(channel_cap)` between worker pairs. `channel_cap = 4096` (mirrors today's spill-buffer size). On overflow, the sender blocks for up to 1 ms then drops the edge to a slow-path `Mutex<VecDeque<>>` "overflow inbox" (rare — only happens if a partition's owner is wedged, which is also a termination-detection bug).

### 3.4 Cluster channels (cross-node case)

Wire `PartitionEdge` / `PartitionEdgeAck` / `RedDfsProbe` / `RedDfsResponse` through the existing `Transport` (`src/distributed/transport.rs`). The framing is unchanged (4-byte length prefix + bincode). The `handler.rs` switch table grows four arms; each arm is symmetric to the existing `StealRequest` / `StealResponse` arms.

### 3.5 Wire-format tests (mirror existing pattern)

Add four `roundtrip_*` tests in `src/distributed/protocol.rs::tests` mirroring the existing `roundtrip_steal_request` style. **These are independent of any TLA+ spec and run on any developer box.**

## 4. Fairness-threshold acceptance — concrete mapping

The original doc handwaved "for our weak/strong fairness shapes the threshold heuristic is equivalent and avoids the product blowup." Concretely:

For each `FairnessConstraint c` (returned from `model.fairness_constraints()`) the acceptance predicate at depth `d` for state `s` on path `π` is:

```text
accept_pending(s, π, c) :=
    match c {
        WeakFairness(action_a) =>
            // depth - last_fired_depth[a] > diameter_threshold
            //   AND action_a was enabled at every state in π since last_fired
            depth(s) - last_fired_depth(action_a, π) > T_wf
            AND action_enabled_at_every_state_since(action_a, π, last_fired)
        StrongFairness(action_a) =>
            // action_a was enabled infinitely often along π but never fired
            depth(s) - last_fired_depth(action_a, π) > T_sf
            AND action_enabled_at_some_state_since(action_a, π, last_fired)
    }
```

Where `T_wf = T_sf = 2 × graph_diameter_estimate`. The diameter estimate starts at 1024 and adapts: every 100K states, recompute as `max(observed_blue_path_depth × 2, current_T)`.

**Soundness gap**: this is a sound *over-approximation* of acceptance — if it fires, there is a path long enough to plausibly violate fairness; the red DFS then proves whether a real cycle exists. **Completeness gap**: if `T` is too small, we miss real violations. The escape hatch: `--liveness-streaming-exploration` gates this; if `T` proves too small for a corpus spec, that spec falls back to the Tarjan path (set the gate to `false` for that spec). The fallback is the same gate that already protects against the BFS-path being unable to handle the spec.

A cleaner long-term fix is the Buchi-product construction; it's left for a follow-on task because it requires a property-LTL parser the runtime does not currently have.

## 5. Termination detection — exact extension

The existing termination logic (`src/runtime.rs:2411-2459`) checks: all workers idle ∧ `worker_queue.has_pending_work() == false` ∧ (cluster mode) `stealer.is_globally_terminated() == true`.

DFS extension: add three predicates to the conjunction.

1. **All DFS stacks empty**: per-worker `dfs_state.stack.is_empty()`. Reuses the existing per-worker idle flag — DFS workers set idle when `stack.is_empty()` and there is no inbound partition edge.
2. **All cross-partition inboxes empty**: per-worker `cross_partition_inbox.is_empty()`. New cache-line-padded `AtomicUsize` per worker, mirroring the existing per-NUMA idle counters.
3. **No in-flight partition messages**: a single `AtomicI64` counter incremented on `PartitionEdge` send and decremented on `PartitionEdgeAck` receive. Mirrors the `pending_steals` counter in `src/distributed/work_stealer.rs:341` exactly. Termination requires this == 0.

The cluster-mode `TerminationToken` round (`src/distributed/protocol.rs:45`) gets two new fields:

```rust
TerminationToken {
    initiator: u32,
    round: u64,
    all_idle: bool,
    /// Sum of all nodes' in-flight partition message counters at the
    /// moment they last forwarded the token.
    inflight_partition_edges: u64,
    /// Sum of all nodes' in-flight red-DFS probes.
    inflight_red_probes: u64,
}
```

Termination requires `all_idle && inflight_partition_edges == 0 && inflight_red_probes == 0` for two consecutive token rounds (the existing protocol's two-round invariant carries over).

## 6. Trace reconstruction — concrete protocol

When red DFS returns `Some(witness_fp)`, the worker assembles:

- **Lasso prefix**: states from blue path index 0 up to `blue_path_fps.iter().position(|&fp| fp == witness_fp)`. These states are still in `dfs_state.stack` (each frame holds the state).
- **Cycle body**: states from the witness index to the top of `blue_path_fps`, plus the red-DFS trail.

When the cycle straddles partitions (some states on this worker's stack, others on a peer's stack), the lasso reporter sends `RedDfsResponse { trail_extension: ... }` carrying the *fps* (not full states) of the peer's portion. The reporter then issues an additional `RequestStateBlob { fps: Vec<u64> }` round-trip to fetch the full states from peers. (New message variant; same framing.) This extra round-trip is acceptable because trace reconstruction is a one-shot operation per violation, not on the hot path.

The full lasso is then written via the existing `trace_minimize.rs` pipeline (which is partition-agnostic — it consumes a flat `Vec<S>`).

## 8. Rollout plan

The original doc said "lift the DFS into the worker hot loop." That's a single-step rollout and it's risky. Concrete staged plan:

1. **Stage 1 (week 1)**: ship `PageAlignedColorMap` + 7 unit tests. No runtime integration. Merge.
2. **Stage 2 (week 2)**: ship the 4 new `Message` variants + round-trip tests + handler stubs that log-and-drop. No runtime integration. Merge.
3. **Stage 3 (week 3)**: ship the DFS worker body behind `--liveness-streaming-exploration`, single-node-only (no cross-partition routing — assert all edges stay in-partition or error). Validate against `wrapper_next_fairness_t1_3` and the 12 fairness specs in the public corpus. Merge if all 12 pass.
4. **Stage 4 (week 4)**: enable cross-partition routing in single-node mode. Validate against same 12 specs. Run the section 7.3 memory benchmark. Merge if >50% RSS reduction confirmed.
5. **Stage 5 (week 5)**: enable cluster-mode cross-partition routing. Run section 7.4 corpus revalidation. Flip the default for fairness specs only after that revalidation is clean. Merge.

Stages 1, 2, 3, 4, 5 are independently mergeable (each stage either lands behind a default-off flag or is a strictly-additive data structure). This avoids the "all-or-nothing" merge risk the original doc accepted.

## 9. What this refinement does not solve

Three open questions remain even with this refinement:

1. **Diameter-threshold completeness**. Section 4's `T_wf = 2 × diameter_estimate` is sound but not complete; a pathological spec with cycle length > 2 × estimated diameter will be missed. Mitigation: the `--liveness-streaming-exploration` flag is opt-in for fairness specs; if it misses a known violation in the corpus, fall back to Tarjan for that spec. A long-term fix needs a Buchi-product construction.
2. **Red DFS termination across partitions**. If a red DFS probe gets stuck cycling between two partitions because each side keeps forwarding to the other, we deadlock. Mitigation: red DFS carries a TTL (`max_red_dfs_hops`, default 1024); on TTL exhaustion it returns `RedDfsTruncated` and the caller falls back to Tarjan for the affected SCC. This is observable in stats.
3. **Trace reconstruction across worker crashes**. If a worker crashes mid-violation-report, the partial blue stack on its side is lost. Mitigation: persist `blue_path_fps` to a worker-local checkpoint at every 10K-frame mark; on crash, the recovery path can rebuild trace prefixes from the checkpoint. Same machinery as the existing `--checkpoint-interval-secs`, just with a smaller granularity.

These are noted here so the next implementer doesn't re-discover them at the bug-surface phase.
