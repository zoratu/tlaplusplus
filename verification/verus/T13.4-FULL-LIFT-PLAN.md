# T13.4 full lift — execution plan

**Status**: planning. The Phase 2 in-place annotation track (`src/storage/verus_smoke.rs`, 38 verified helpers / 177 wired sites / 31 files) is complete and shipping at v1.2.7. This document plans the *full* lift: replacing the unsafe pointer arithmetic in `src/storage/page_aligned_fingerprint_store.rs::FingerprintShard` with a `Vec<PAtomicU64>` + `Tracked<Map<int, PermissionU64>>` design that admits direct method-level `requires` / `ensures` verification.

The premises (verification feasibility of every pattern the lift touches) are already proven in 195 verified items across 11 standalone artifacts — see `T13.4-PHASE2-CLOSURE.md` for the per-gap blueprint pointers.

## Lift target

The `FingerprintShard` hot-path methods today are unsafe over an mmap'd region:

```rust
fn contains(&self, fp: u64) -> bool {
    let table = self.table.load(Ordering::Acquire);
    let capacity = self.capacity.load(Ordering::Acquire);
    let mut index = (fp as usize) % capacity;
    loop {
        let entry = unsafe { (*table.add(index)).load(Ordering::Acquire) };
        if entry == 0 { return false; }
        if entry == fp { return true; }
        index = (index + 1) % capacity;
    }
}
```

The verified shadow in `shard_methods.rs::probe_slot_for_contains` shows the target shape:

```rust
pub fn probe_slot_for_contains(
    cells: &ShardCells,
    idx: usize,
    fp: u64,
    Tracked(perm): Tracked<&PermissionU64>,
) -> (result: ProbeStep)
    requires
        cells.slots.len() > 0,
        idx < cells.slots.len(),
        fp != empty_slot(),
        perm.view().patomic == cells.slots@[idx as int].id(),
    ensures /* ... abstract semantics tied to tier-A's tab_lookup ... */
```

The lift threads `Vec<PAtomicU64>` + a per-slot tracked permission map through every read/write call site.

## Lift strategy — wrapper-then-cutover

The current `FingerprintShard` is hot-path critical for the runtime. A direct in-place rewrite would break every caller atomically. The pragmatic path is:

### Phase A — Wrapper carve-out (low risk, no behavior change)

Introduce `src/storage/verified_fingerprint_shard.rs::VerifiedShard` — a parallel struct gated behind a new `verified-fp-shard` cargo feature (default off):

- Uses `Vec<PAtomicU64>` + `Tracked<Map<int, PermissionU64>>` per the shadow blueprint.
- Implements the same public API as `FingerprintShard` (`new`, `contains`, `contains_or_insert`, `len`, `load_factor`, ...).
- Each method body is a `verus!{}` block annotated against tier-A spec predicates.
- Exists alongside `FingerprintShard` — not used by the runtime in Phase A.

Validation: a dedicated `tests/verified_fp_shard_equivalence.rs` exercises both stores side-by-side on a uniform fingerprint stream and asserts identical behavior (contains/insert results, load factor at each step).

PR-sized chunks within Phase A:

1. **A.1** — `VerifiedShard::new`, `len`, `load_factor`, `capacity`. No probe-loop yet. Validates the Vec<PAtomicU64> + Tracked permission map construction.
2. **A.2** — `VerifiedShard::contains` (read-only probe). Reuses the `probe_slot_for_contains` shadow's contract.
3. **A.3** — `VerifiedShard::contains_or_insert` (CAS-insert probe). Reuses `cas_insert_or_observe`.
4. **A.4** — `VerifiedShard::rehash_batch_counted` (resize migration). Reuses `rehash_one_step`.
5. **A.5** — Resize coordination (seqlock toggle + new/old table swap). Two paths:
   - (preferred) Lock-free atomic-pointer swap per `atomic_ptr_with_epoch.rs` (Tier A.9 gap-1 PoC).
   - (fallback) `vstd::rwlock::RwLock<VerifiedShard, WellFormedShardPred>` per `shard_exec_wired.rs` (Tier A.10).

### Phase B — Behind-flag swap

After Phase A passes equivalence tests on the full corpus:

- Add an `EngineConfig::use_verified_fp_shard: bool` field, default `false`.
- At runtime startup, if the flag is set, instantiate `VerifiedShard` instead of `FingerprintShard`.
- Existing callers (`worker.rs`, batch APIs, S3 persistence) interact with both via a small trait `FingerprintStoreShard` extracted from the current `FingerprintShard` API. Each method's signature stays unchanged; the trait is a no-op for the existing struct.
- Corpus run + diff_tlc gate validates no behavior regression with the flag on.

PR-sized chunks within Phase B:

1. **B.1** — Extract `trait FingerprintStoreShard` from `FingerprintShard`'s public API. Default impl uses the existing struct. No behavior change.
2. **B.2** — Implement the trait on `VerifiedShard`. Both impls coexist.
3. **B.3** — Wire `EngineConfig::use_verified_fp_shard` and the runtime branch.
4. **B.4** — Run full corpus + diff_tlc + chaos smoke with the flag on. Capture verdict equivalence.

### Phase C — Default cutover

Once Phase B has shipped two release cycles without regressions:

- Flip `EngineConfig::use_verified_fp_shard` default to `true`.
- Update CLAUDE.md to reflect that the verified shard is the production path; `FingerprintShard` remains as the unverified-fallback impl.
- Optionally: delete the legacy `FingerprintShard` in a follow-up release (only after at least one full release of the verified default + long-soak run).

## Caller-threading scope (step 5)

The reach of `Tracked<...>` parameters depends on where the verified boundary lands. Two extremes:

**Conservative**: keep `Tracked<...>` inside `VerifiedShard`'s methods only. Public API stays `&self`; the tracked permissions live in a member field (e.g. `Tracked<Map<int, PermissionU64>>`) and the wrapper threads them internally. Pattern: see `shard_exec_wired.rs::read` using `Shared<AtomicInvariant<GhostStuff>>`.

**Maximal**: `Tracked<&mut Permissions>` parameters surface in the public API. Every caller in `runtime.rs`, `work_stealing_queues.rs`, batch APIs, S3 persistence must explicitly pass the permission. Blast radius: ~50-80 call sites across 8-12 files.

Recommended: start with **conservative** (Phase A through B). The wrapper carries the permission internally via `vstd::shared::Shared` or an `AtomicInvariant`. Maximal threading is a future-decision item, only worth doing if there's a specific safety property the public-API permission would enable (e.g. compile-time enforcement that callers never alias the table while a resize is in progress).

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| Vec<PAtomicU64> performance vs raw mmap pointer arithmetic | Phase B corpus + chaos smoke captures any throughput regression. Bound: expect ≤10% slowdown vs raw pointers; if larger, fall back to AtomicPtr<PAtomicU64> array. |
| Resize protocol complexity grows under tracked-permission threading | The shadow `atomic_ptr_with_epoch.rs` (Tier A.9) already proves the lock-free RCU + counting-permission pattern. Use it. |
| Tracked-permission overhead in CAS hot loop | Verus's tracked items are erased at runtime — zero overhead. Validated by the cargo-verus-demo at `verification/cargo-verus-demo/`. |
| API divergence between `FingerprintShard` and `VerifiedShard` accumulates | The `FingerprintStoreShard` trait (B.1) freezes the API contract. New methods land on the trait first. |
| Behind-flag verification gap: `--features verus` doesn't exercise `VerifiedShard` until flag is on | Phase A's equivalence tests run unconditionally; verus check runs on every commit via the existing CI gate. |

## What this plan does *not* commit to

- Deleting `FingerprintShard` (Phase C optional, follow-up).
- Maximal Tracked-API threading (deferred per "Caller-threading scope" above).
- Lock-free vs RwLock for resize coordination (the choice is Phase A.5 — both are verified; pick whichever simplifies the rest of the wrapper).

## Acceptance criteria

The lift is complete when:

1. `VerifiedShard` passes the equivalence tests in `tests/verified_fp_shard_equivalence.rs` on every commit.
2. `cargo verus check --features verus` includes the verified-shard methods at top-level (not just helpers).
3. Default runtime instantiation uses `VerifiedShard` (Phase C).
4. The full TLA+ corpus passes with `use_verified_fp_shard=true` (verdicts + state counts match the legacy path).
5. The 1-hour chaos soak passes with the verified shard active.
