# T13.2 + T13.4 — production-code Verus annotation design

Status: **design doc only, no production-code rewrite shipped.**
Outcome: **Path B** (per the T13.2/T13.4 landing brief). This document
lays out the precise refactor that a future agent-week or research run
should execute, along with the concrete capability gaps that make the
in-place annotation unworkable today.

The companion artifact is `verification/verus/shard_methods.rs`
(17 verified, ~0.7s) — the production-shape shadow that this design
extrapolates to a full production-code rewrite.

## Decision and rationale

Three fundamental incompatibilities between the production
`FingerprintShard` shape and Verus's `vstd` capability surface together
make the in-place rewrite path unworkable inside the 12-hour budget
**without** either (a) introducing `#[verifier::external_body]` axioms
to paper over the hard parts (explicitly forbidden by the brief), or
(b) rewriting the production data layout in ways that cost measurable
runtime performance and burn the brief's "must not break `cargo build
--release`" gate. They are documented under "Capability gap" below;
the upshot is that an honest Path-A landing is multi-week research
work, not a 12-hour timebox, and the shadow file already supplies the
working blueprint.

The shadow file already verifies the *method bodies* against the
tier-A `cas_step`/`tab_lookup`/`tab_insert` semantics. The remaining
work — and where the gaps bite — is in the shard *structure* and the
*allocation/swap* protocol.

## Ghost state

Add to `FingerprintShard`:

```rust
#[cfg(verus_verify)]
pub tracked table_perm: Tracked<TableState>,
```

where `TableState` (verus-only) is:

```rust
verus! {
pub tracked struct TableState {
    pub points_to: PointsToArray<HashTableEntry>,
    pub capacity: nat,
    pub generation: nat,            // bumped on every successful resize swap
    pub seq_value: nat,             // mirrors self.seq.load() at last sync point
    pub old_points_to: Option<PointsToArray<HashTableEntry>>,
    pub new_points_to: Option<PointsToArray<HashTableEntry>>,
}
}
```

Invariant `cells_wf` (analogous to the shadow's): every slot in
`points_to` is paired with the implicit per-slot permission projected
from the `PointsToArray`, capacity matches the published
`capacity: AtomicUsize`, generation parity matches `seq_value`.

To make the invariant maintainable across `&self` calls without
linear-permission threading at every call-site, the design uses
**vstd's `AtomicInvariant<TableState>`** — a Verus primitive that wraps
ghost state behind an internal lock-free invariant. Each shard method
opens the invariant for the duration of its operation, performs the
load/CAS/store with the appropriate ghost permission, and re-closes the
invariant. This is the canonical Verus pattern for "ghost state
attached to a `Send + Sync` struct accessed via `&self`."

## The annotated shape (skeleton)

```rust
verus! {

impl FingerprintShard {
    pub fn contains(&self, fp: u64) -> (b: bool)
        ensures
            // Soundness: if true, the abstract table contains fp.
            b ==> self.abstract_view().contains(fp_or_one(fp)),
    {
        let fp = if fp == 0 { 1 } else { fp };
        loop
            invariant
                fp != 0,
                self.cells_wf(),
        {
            let seq_before = self.seq.load(Ordering::Acquire);
            if seq_before % 2 == 1 {
                // resize-mode path: identical to production lines 562-622,
                // but every `unsafe { from_raw_parts(ptr, cap) }` is replaced
                // by `open_atomic_invariant!(...)` that yields a ghost
                // `&PointsToArray` and lets us invoke `PAtomicU64::load`
                // through the per-slot permission.
                ...
            }
            // normal-path probe loop with `Tracked<&PointsToArray>`:
            open_atomic_invariant!(&self.table_perm => state => {
                let cap = state.capacity;
                let mut index = (fp as usize) % cap;
                let mut probes = 0u64;
                while probes < cap as u64
                    invariant
                        fp != 0,
                        index < cap,
                        probes <= cap as u64,
                        state.cells_wf(),
                    decreases cap as u64 - probes,
                {
                    let perm = state.points_to.borrow_slot(index as int);
                    let stored = ATOMIC_LOAD(&entry_at(&state, index).fp, perm);
                    if stored == fp { return true; }
                    if stored == 0 { break; }
                    index = (index + 1) % cap;
                    probes += 1;
                }
            });
            let seq_after = self.seq.load(Ordering::Acquire);
            if seq_before == seq_after { return false; }
        }
    }

    pub fn contains_or_insert(&self, fp: u64, stats: &FingerprintStatsAtomic)
        -> (b: bool)
        ensures
            !b ==> self.abstract_view().contains(fp_or_one(fp)),
            // monotonicity
            self.abstract_view().subset_of(old(self).abstract_view().insert(fp_or_one(fp))),
    {
        // body: same shape as `contains`, but the inner loop opens the
        // invariant *mutably* on the empty-slot branch to invoke
        // `PAtomicU64::compare_exchange`. After CAS succeeds, the
        // ghost state's `points_to.update(index, fp)` is performed
        // inside the same invariant scope.
        ...
    }
}

} // verus!
```

The Verus tactic burden is concentrated in three places:

1. **Probe-loop invariant.** Tier A's `lemma_probe_terminus_bounded`
   already discharges termination at the spec level. Lifting it
   into an exec loop requires writing
   `invariant probes <= cap as u64 && index == probe_index(fp, probes, cap)`
   plus a `decreases cap as u64 - probes` clause. Estimated 1-2 days
   of Verus tactic engineering.
2. **CAS-step ghost update.** The shadow's
   `lemma_cas_inserted_matches_tier_a` gives the proof that the abstract
   `slots_view` advances to `update(idx, fp)`. Inside the production
   `contains_or_insert`, after a successful CAS, we must invoke this
   lemma against `state.points_to` and refresh
   `state.points_to = state.points_to.update(...)`. Estimated 1 day.
3. **Seqlock retry loop.** The outer `loop { let seq_before = ...;
   ...; if seq_before == seq_after { return false } }` loop has no
   natural `decreases` measure (it terminates because resizes are
   bounded in any concrete execution, but not in the spec). Tier A's
   `lemma_reader_terminates` proves this with a `decreases R - i`
   clause where `R` is the number of resizes the writer performs
   during the reader's lifetime. Lifting this into an exec loop
   requires either (a) accepting `loop` without `decreases` and
   marking the function `#[verifier::not_terminating]`, or (b)
   tracking `R` as a ghost invariant and threading a "writer's resize
   count" decrement through `finalize_resize`. Estimated 2-3 days.

Total tactic engineering on top of the shadow: roughly 4-6 agent-days
*if* the capability gaps below didn't exist. They do.

## Capability gap: why this isn't a 12-hour landing

### Gap 1: `AtomicPtr<HashTableEntry>` table swap

Production `FingerprintShard.table` is `AtomicPtr<HashTableEntry>`
(line 130). Resize swaps it atomically (line 368:
`self.table.store(new_table, Ordering::Release)`). The pointer that
the field holds *changes* across resizes, while the per-slot atomics
inside the *previous* allocation must remain accessible to in-flight
readers (they're parked in `old_table` for the duration of the rehash,
freed via `cleanup_old_memory` only after at least one full resize
cycle has elapsed; line 545).

Verus's `vstd::raw_ptr::PPtr<T>` and the underlying `PointsToRaw`
permission both have a fixed `id() : Ghost<int>` that names a *single*
allocation. There is no Verus type for "an atomic that holds a
PPtr-equivalent which can be swapped to a different allocation, with
overlapping lifetimes for the old and new permissions." The closest
fit (`AtomicInvariant<...>` parameterised on `TableState` with
`old_points_to: Option<PointsToArray<...>>` and `new_points_to:
Option<PointsToArray<...>>`) requires hand-coding the lifetime
relation between three concurrent allocations: `old`, `current`, `new`.
Verus does not yet provide a discharge tactic for that relation — the
reasoning is research-grade (see "linear types for swappable
references" literature, Beillahi et al., POPL 2021).

The honest workaround is `#[verifier::external_body]` on every method
that touches `self.table.load(...)` followed by
`from_raw_parts(...)` — which is exactly what the brief forbids.

### Gap 2: `mmap(MAP_HUGETLB | MAP_POPULATE)` allocation

Production lines 187-256 (`FingerprintShard::new`) call
`allocate_huge_pages` or `allocate_file_backed`, each of which calls
`mmap()` from `libc`. Verus's `vstd::raw_ptr::PPtr::new` mints a
`PointsToRaw` permission for *Verus-allocated heap* (typically
`Box::new` or `vec!`-backed). There is no public Verus API to mint a
fresh `PointsToArray<T>` permission for an mmap'd region.

This is structural: Verus's allocation model assumes the global
allocator owns all memory. mmap'd anonymous regions live outside that
model. The standard escape hatch is `#[verifier::external_body]` on
the allocation site — which is acceptable per Verus's own coding
guidelines for FFI primitives, but the *result type* cannot be
`Tracked<PointsToArray<HashTableEntry>>` because Verus refuses to
admit a fresh ghost token from an external-body function (it would
let users forge permissions). The accepted workaround is a hand-coded
"memory provenance axiom" — which is again precisely what the brief
forbids.

### Gap 3: `&self`-callable methods with linear ghost permissions

The `FingerprintShard` is shared across worker threads behind an `Arc`
(line 178: `unsafe impl Send for FingerprintShard {}`). Every method
takes `&self`. In Verus, `Tracked<PointsToArray<T>>` is a *linear*
ghost: it cannot be borrowed via `&self` without going through an
invariant.

The standard pattern is `vstd::invariant::AtomicInvariant<S>` where
`S` carries the linear ghost. Each method
`open_atomic_invariant!(&self.field => state => { ... })` opens the
invariant, gets `&mut S` for the duration of the block, and re-closes.

This works, but it imposes two real costs on the production code:

1. **The invariant `inv: spec_fn(S) -> bool` must be a closed predicate
   over `S`.** Our `cells_wf` invariant references the per-slot
   permissions inside `points_to` — fine. But it must *also* tie
   `state.capacity` to `self.capacity.load()`, which is an *atomic*,
   not part of `S`. Crossing that boundary requires a *second*
   AtomicInvariant for the seqlock counter, plus a tactic that the two
   invariants compose. Verus does not yet have a "two-phase
   invariant" discharge primitive; the closest is a manual `open
   _atomic_invariant!` nested 2-deep, which serializes invariant access
   and would *measurably regress* the benchmarked 220K states/sec
   throughput.

2. **`open_atomic_invariant!` is a syntactic block with bounded
   duration.** Production code's `contains_or_insert` (lines 658-866)
   loops up to `MAX_RETRIES_BEFORE_PANIC = 1_000_000` iterations,
   yielding to the OS scheduler periodically. Holding an invariant
   open across `std::thread::yield_now()` (line 678) is forbidden by
   the invariant discipline (the invariant must be re-closeable on
   exit, but yielding hands control to other threads which may
   themselves try to open the same invariant). Restructuring the
   retry loop to open-close the invariant per-iteration is feasible
   but adds an `acquire(invariant)` overhead at every probe step —
   the production hot path executes ~10-100M probe steps per second
   per worker; even a 10ns invariant open/close cost per step
   converts the headline 220K states/sec benchmark into ~50K-100K.
   That violates the brief's "must keep `cargo build --release`
   working AND not regress perf" gate.

The shadow file (`shard_methods.rs`) sidesteps this by taking
`Tracked<&PermissionU64>` as a *function parameter*, expecting the
caller to have already done the invariant open. That pattern works
inside Verus but would force every caller of
`FingerprintShard::contains_or_insert` (the runtime worker loop, the
batch APIs at lines 1507-1530) to also be Verus-annotated and to also
open the invariant. The blast radius reaches `runtime.rs`,
`work_stealing_queues.rs`, and most of the `storage/` module.

## What a fully-verified production code path *would* look like

Concretely: **a Verus `state_machines!` reformulation of the
fingerprint store** (Beillahi 2021, AWS s2n-quic verification work,
Hance et al. "Storage Systems are Distributed Systems" SOSP 2020).
This is the same v1.2.0+ research path called out for T13.5's
unbounded-fairness reader liveness in `reader_liveness.rs`.

In a `state_machines!` formulation:
- The whole shard becomes a state machine `FingerprintShardSM` with
  transitions `insert(fp)`, `begin_resize`, `rehash_one(idx)`,
  `finalize_resize`, `read(fp)`.
- The pointer-swap and mmap allocation drop out — they're physical
  realisations of the abstract `begin_resize` and `finalize_resize`
  transitions.
- Linear ghosts are tied to *transitions*, not to method calls, so
  the `&self` / linear-token tension dissolves.
- `state_machines!` natively supports temporal invariants and
  liveness, so the seqlock retry loop's reader liveness and the
  CAS soundness get proved in the same framework.

Estimated cost: 5-7 agent-weeks for a full port (matches the
T13.5-full-discharge estimate documented in
`verification/verus/README.md` line 219). Best done jointly with the
T13.5 liveness-axiom discharge since both share the same machinery.

## Concrete next-step checklist

If a future agent run takes this on:

1. **Pick A vs B for the table-swap problem.** Either (a) accept a
   single `#[verifier::external_body]` on `FingerprintShard::new` and
   on the resize swap (compromises gap 1 + gap 2 minimally) — but the
   brief explicitly forbids this approach, so really (b) port to
   `state_machines!`. Allow 2-3 weeks for (b).
2. **Pick C vs D for the per-slot permission threading.** Either
   (c) `AtomicInvariant` per shard with closed `S = TableState` and
   accept the perf hit, OR (d) thread `Tracked<...>` parameters
   through `contains_or_insert` and propagate to the runtime. (d) has
   smaller perf cost but blast-radius reaches `runtime.rs`.
3. **Land the probe-loop invariant first** (1-2 days, no new Verus
   capability needed). This is the smallest standalone win and would
   convert tier-A's spec-level `lemma_probe_terminus_bounded` into an
   exec-level proof obligation discharged by the Rust compiler. Even
   under Path B, this is shippable as a v1.2.0 deliverable.
4. **Then land CAS-step ghost update** (1 day, builds on probe-loop
   invariant). Brings the shadow's `lemma_cas_inserted_matches_tier_a`
   into the production CAS site.
5. **Then revisit table-swap.** By this point, steps 1-2 will have
   forced a decision on the framework (`state_machines!` vs in-place).
6. **Validate continuously.** Every commit must keep `cargo build
   --release`, `cargo test --release` (756+ tests),
   `cargo test --release --features failpoints` (776+ tests), and
   `cargo test --release --features symbolic-init` (774+ tests) green.
