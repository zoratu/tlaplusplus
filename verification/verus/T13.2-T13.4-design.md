## Decision and rationale

Three fundamental incompatibilities between the `FingerprintShard` itself shape and Verus's `vstd` capability surface together make the in-place rewrite path unworkable **without** either (a) introducing `#[verifier::external_body]` axioms to paper over the hard parts, or (b) rewriting the shipping data layout in ways that cost measurable runtime performance and break the "must not break `cargo build --release`" gate. They are documented under "Capability gap" below; the upshot is that an honest Path-A landing is open-ended work, and the shadow file already supplies the working blueprint.

The shadow file already verifies the *method bodies* against the tier-A `cas_step`/`tab_lookup`/`tab_insert` semantics. The remaining work — and where the gaps bite — is in the shard *structure* and the *allocation/swap* protocol.

## Ghost state

Add to `FingerprintShard`:

```rust
#[cfg(verus_verify)]
pub tracked table_perm: Tracked<TableState>,
```

where `TableState` (verus-only) is:

```rust
verus! {
pub tracked struct TableState {
    pub points_to: PointsToArray<HashTableEntry>,
    pub capacity: nat,
    pub generation: nat,            // bumped on every successful resize swap
    pub seq_value: nat,             // mirrors self.seq.load() at last sync point
    pub old_points_to: Option<PointsToArray<HashTableEntry>>,
    pub new_points_to: Option<PointsToArray<HashTableEntry>>,
}
}
```

Invariant `cells_wf` (analogous to the shadow's): every slot in `points_to` is paired with the implicit per-slot permission projected from the `PointsToArray`, capacity matches the published `capacity: AtomicUsize`, generation parity matches `seq_value`.

To make the invariant maintainable across `&self` calls without linear-permission threading at every call-site, the design uses **vstd's `AtomicInvariant<TableState>`** — a Verus primitive that wraps ghost state behind an internal lock-free invariant. Each shard method opens the invariant for the duration of its operation, performs the load/CAS/store with the appropriate ghost permission, and re-closes the invariant. This is the canonical Verus pattern for "ghost state attached to a `Send + Sync` struct accessed via `&self`."

## The annotated shape (skeleton)

```rust
verus! {

impl FingerprintShard {
    pub fn contains(&self, fp: u64) -> (b: bool)
        ensures
            // Soundness: if true, the abstract table contains fp.
            b ==> self.abstract_view().contains(fp_or_one(fp)),
    {
        let fp = if fp == 0 { 1 } else { fp };
        loop
            invariant
                fp != 0,
                self.cells_wf(),
        {
            let seq_before = self.seq.load(Ordering::Acquire);
            if seq_before % 2 == 1 {
                // resize-mode path: identical to lines 562-622,
                // but every `unsafe { from_raw_parts(ptr, cap) }` is replaced
                // by `open_atomic_invariant!(...)` that yields a ghost
                // `&PointsToArray` and lets us invoke `PAtomicU64::load`
                // through the per-slot permission.
                ...
            }
            // normal-path probe loop with `Tracked<&PointsToArray>`:
            open_atomic_invariant!(&self.table_perm => state => {
                let cap = state.capacity;
                let mut index = (fp as usize) % cap;
                let mut probes = 0u64;
                while probes < cap as u64
                    invariant
                        fp != 0,
                        index < cap,
                        probes <= cap as u64,
                        state.cells_wf(),
                    decreases cap as u64 - probes,
                {
                    let perm = state.points_to.borrow_slot(index as int);
                    let stored = ATOMIC_LOAD(&entry_at(&state, index).fp, perm);
                    if stored == fp { return true; }
                    if stored == 0 { break; }
                    index = (index + 1) % cap;
                    probes += 1;
                }
            });
            let seq_after = self.seq.load(Ordering::Acquire);
            if seq_before == seq_after { return false; }
        }
    }

    pub fn contains_or_insert(&self, fp: u64, stats: &FingerprintStatsAtomic)
        -> (b: bool)
        ensures
            !b ==> self.abstract_view().contains(fp_or_one(fp)),
            // monotonicity
            self.abstract_view().subset_of(old(self).abstract_view().insert(fp_or_one(fp))),
    {
        // body: same shape as `contains`, but the inner loop opens the
        // invariant *mutably* on the empty-slot branch to invoke
        // `PAtomicU64::compare_exchange`. After CAS succeeds, the
        // ghost state's `points_to.update(index, fp)` is performed
        // inside the same invariant scope.
        ...
    }
}

} // verus!
```

The Verus tactic burden is concentrated in three places:

1. **Probe-loop invariant.** Tier A's `lemma_probe_terminus_bounded` already discharges termination at the spec level. Lifting it into an exec loop requires writing `invariant probes <= cap as u64 && index == probe_index(fp, probes, cap)` plus a `decreases cap as u64 - probes` clause.
2. **CAS-step ghost update.** The shadow's `lemma_cas_inserted_matches_tier_a` gives the proof that the abstract `slots_view` advances to `update(idx, fp)`. Inside the `contains_or_insert`, after a successful CAS, we must invoke this lemma against `state.points_to` and refresh `state.points_to = state.points_to.update(...)`.
3. **Seqlock retry loop.** The outer `loop { let seq_before = ...; ...; if seq_before == seq_after { return false } }` loop has no natural `decreases` measure (it terminates because resizes are bounded in any concrete execution, but not in the spec). Tier A's `lemma_reader_terminates` proves this with a `decreases R - i` clause where `R` is the number of resizes the writer performs during the reader's lifetime. Lifting this into an exec loop requires either (a) accepting `loop` without `decreases` and marking the function `#[verifier::not_terminating]`, or (b) tracking `R` as a ghost invariant and threading a "writer's resize count" decrement through `finalize_resize`.

The tactic engineering on top of the shadow is tractable *if* the capability gaps below didn't exist. They do.

## Capability gap

### Gap 1: `AtomicPtr<HashTableEntry>` table swap

The `FingerprintShard.table` is `AtomicPtr<HashTableEntry>` (line 130). Resize swaps it atomically (line 368: `self.table.store(new_table, Ordering::Release)`). The pointer that the field holds *changes* across resizes, while the per-slot atomics inside the *previous* allocation must remain accessible to in-flight readers (they're parked in `old_table` for the duration of the rehash, freed via `cleanup_old_memory` only after at least one full resize cycle has elapsed; line 545).

Verus's `vstd::raw_ptr::PPtr<T>` and the underlying `PointsToRaw` permission both have a fixed `id() : Ghost<int>` that names a *single* allocation. There is no Verus type for "an atomic that holds a PPtr-equivalent which can be swapped to a different allocation, with overlapping lifetimes for the old and new permissions." The closest fit (`AtomicInvariant<...>` parameterised on `TableState` with `old_points_to: Option<PointsToArray<...>>` and `new_points_to: Option<PointsToArray<...>>`) requires hand-coding the lifetime relation between three concurrent allocations: `old`, `current`, `new`. Verus does not yet provide a discharge tactic for that relation — the reasoning is open-ended (see "linear types for swappable references" literature, Beillahi et al., POPL 2021).

The honest workaround is `#[verifier::external_body]` on every method that touches `self.table.load(...)` followed by `from_raw_parts(...)` — which is exactly what the brief forbids.

#### Gap 1 update (2026-05-13): no new primitive needed

This gap was filed upstream as [verus-lang/verus#2437](https://github.com/verus-lang/verus/issues/2437). Core Verus collaborator @tjhance responded that the pattern is verifiable today via the **Storage Protocol** machinery in `tokenized_state_machine!` (a.k.a. VerusSync), using the same **counting permissions** pattern demonstrated in `verus/examples/state_machines/counting.rs`. No new `vstd` primitive is required.

We landed `verification/verus/atomic_ptr_with_epoch.rs` (16 verified items, 0 errors on Verus 0.2026.05.13 / aarch64) as a proof-of-concept skeleton:

- `EpochProtocol<T>` is a `tokenized_state_machine!` modelling the single-epoch RCU swap pattern: a `stored: Option<T>` slot (storage_option) holding the linear permission, a `main_counter: Option<(nat, T)>` (variable) tracking outstanding readers, and a `read_ref: Multiset<T>` (multiset) minting one token per live reader.
- Five proof-level operations: `new()`, `publish()` (deposit the permission, go from empty to "(0, t)"), `reclaim()` (withdraw the permission, requires `count == 0`), `acquire_read()` (increment count, mint a `ReadRef`), `release_read()` (decrement count, consume `ReadRef`).
- One `guard` property: `read_guards(&ReadRef) -> &T` lets a reader borrow `&T` through a live `ReadRef` without consuming the permission — the lifetime-based mechanism @tjhance pointed at.
- Invariant `main_inv` proves: the multiset's size equals the counter, and every token names the same `T` as is stored.

#### Cross-survey of Verus examples (2026-05-13)

After landing the PoC we surveyed the rest of the Verus example tree to confirm we hadn't missed an existing template before posting a follow-up question upstream. Findings (all positive — no further upstream asks needed for the patterns we were worried about):

1. **`examples/state_machines/arc.rs` (331 lines) is the multi-reader RCU template, fully exec-wired.** It defines `RefCounter<Perm>` with exactly the same three sharded fields as our PoC (`storage_option` for the linear permission, `variable` counter, `multiset` reader tokens) plus `do_clone` / `dec_basic` / `dec_to_zero` transitions, and wires the protocol to a real `PAtomicU64` refcount + `PPtr<InnerArc<S>>` allocation. The exec methods `MyArc::new` / `borrow` / `clone` / `dispose` show the full pattern, including `open_atomic_invariant!` around each `compare_exchange_weak` site. **This is a working template for the FingerprintShard exec wiring** — Arc-of-Arc style: each epoch's allocation is a separate `RefCounter` instance; swap publishes a new instance, the old one stays live until its readers drain.

2. **`examples/scache/rwlock.rs` (1180 lines) is the heavyweight template.** Tokenized state machine for a TLB-cache rwlock with shared / exclusive / loading / writeback / claimed states across multiple buckets — far more complex than `FingerprintShard` needs, but proves the pattern scales to industrial-strength concurrent data structures.

3. **`vstd::raw_ptr::allocate` is itself `#[verifier::external_body]` and mints fresh `Tracked<PointsToRaw>` tokens via `Tracked::assume_new()`.** Returns `(*mut u8, Tracked<PointsToRaw>, Tracked<Dealloc>)` with `ensures pt.1@.is_range(pt.0.addr() as int, size as int)`. This is precisely the pattern gap 2 (below) claimed Verus refuses — and the claim was wrong. Verus admits a fresh ghost token from an external-body function any time the post-condition is hand-discharged at the FFI boundary. Same template applies to mmap. See the updated gap-2 section below.

4. **`vstd::logatom.rs` provides the logically-atomic specification framework.** Traits `ReadOperation` / `MutOperation` / `ReadLinearizer` / `MutLinearizer` lift the proof obligation out of the hot path: the exec function takes a linearizer as a ghost parameter, the linearization point is abstracted internally, and there is no per-step `open_atomic_invariant!`. This is the modern Iris-style alternative to gap 3's concern (below). See the updated gap-3 section.

5. **`vstd::shared::Shared<T>`** (used in `arc.rs::MyArc::inv`) is the Arc-like sharing wrapper for distributing an `AtomicInvariant` across worker clones. Sidesteps the bare `Arc<...>` + linear-ghost incompatibility.

6. **`struct_with_invariants!`** declares an exec struct, its `wf` predicate, and the binding to its `AtomicInvariant` in a single block.

### Gap 2: `mmap(MAP_HUGETLB | MAP_POPULATE)` allocation

Lines 187-256 (`FingerprintShard::new`) call `allocate_huge_pages` or `allocate_file_backed`, each of which calls `mmap()` from `libc`. Verus's `vstd::raw_ptr::PPtr::new` mints a `PointsToRaw` permission for *Verus-allocated heap* (typically `Box::new` or `vec!`-backed). There is no public Verus API to mint a fresh `PointsToArray<T>` permission for an mmap'd region.

This is structural: Verus's allocation model assumes the global allocator owns all memory. mmap'd anonymous regions live outside that model. The standard escape hatch is `#[verifier::external_body]` on the allocation site — which is acceptable per Verus's own coding guidelines for FFI primitives, but the *result type* cannot be `Tracked<PointsToArray<HashTableEntry>>` because Verus refuses to admit a fresh ghost token from an external-body function (it would let users forge permissions). The accepted workaround is a hand-coded "memory provenance axiom" — which is again precisely what the brief forbids.

#### Gap 2 update (2026-05-13): the original framing was wrong

The claim that "Verus refuses to admit a fresh ghost token from an external-body function" is **factually incorrect**. `vstd::raw_ptr::allocate` (source/vstd/raw_ptr.rs line 907) is the canonical example: it is `#[verifier::external_body]`, wraps `alloc::alloc::alloc(layout)`, and returns `(*mut u8, Tracked<PointsToRaw>, Tracked<Dealloc>)` with the tracked permissions minted via `Tracked::assume_new()` and the post-condition `ensures pt.1@.is_range(pt.0.addr() as int, size as int)` discharged at the FFI boundary by trust. The same pattern works for mmap:

```rust
#[verifier::external_body]
pub fn mmap_allocate_huge_pages(size: usize) -> (pt: (
    *mut u8,
    Tracked<PointsToRaw>,
    Tracked<MmapDealloc>,  // analog of Dealloc, parameterised on munmap shape
))
    requires
        size != 0,
        size % (2 * 1024 * 1024) == 0,  // huge-page alignment
    ensures
        pt.1@.is_range(pt.0.addr() as int, size as int),
        pt.1@.provenance() == pt.0@.provenance,
    opens_invariants none
{
    let p = unsafe { libc::mmap(
        core::ptr::null_mut(),
        size,
        libc::PROT_READ | libc::PROT_WRITE,
        libc::MAP_ANONYMOUS | libc::MAP_PRIVATE | libc::MAP_HUGETLB | libc::MAP_POPULATE,
        -1, 0,
    ) };
    if p == libc::MAP_FAILED { std::process::abort(); }
    (p as *mut u8, Tracked::assume_new(), Tracked::assume_new())
}
```

The trust assumption is identical to `vstd::raw_ptr::allocate`: the implementation actually does what the `ensures` clause says. We're not adding a fundamentally new axiom — we're extending the same `GlobalAlloc` trust boundary to mmap. Gap 2 is design and engineering, not a vstd capability gap.

### Gap 3: `&self`-callable methods with linear ghost permissions

The `FingerprintShard` is shared across worker threads behind an `Arc` (line 178: `unsafe impl Send for FingerprintShard {}`). Every method takes `&self`. In Verus, `Tracked<PointsToArray<T>>` is a *linear* ghost: it cannot be borrowed via `&self` without going through an invariant.

The standard pattern is `vstd::invariant::AtomicInvariant<S>` where `S` carries the linear ghost. Each method `open_atomic_invariant!(&self.field => state => { ... })` opens the invariant, gets `&mut S` for the duration of the block, and re-closes.

This works, but it imposes two real costs on the shipping code:

1. **The invariant `inv: spec_fn(S) -> bool` must be a closed predicate over `S`.** Our `cells_wf` invariant references the per-slot permissions inside `points_to` — fine. But it must *also* tie `state.capacity` to `self.capacity.load()`, which is an *atomic*, not part of `S`. Crossing that boundary requires a *second* AtomicInvariant for the seqlock counter, plus a tactic that the two invariants compose. Verus does not yet have a "two-phase invariant" discharge primitive; the closest is a manual `open _atomic_invariant!` nested 2-deep, which serializes invariant access and would *measurably regress* the benchmarked 220K states/sec throughput.

2. **`open_atomic_invariant!` is a syntactic block with bounded duration.** Shipping code's `contains_or_insert` (lines 658-866) loops up to `MAX_RETRIES_BEFORE_PANIC = 1_000_000` iterations, yielding to the OS scheduler periodically. Holding an invariant open across `std::thread::yield_now()` (line 678) is forbidden by the invariant discipline (the invariant must be re-closeable on exit, but yielding hands control to other threads which may themselves try to open the same invariant). Restructuring the retry loop to open-close the invariant per-iteration is feasible but adds an `acquire(invariant)` overhead at every probe step — the shipping hot path executes ~10-100M probe steps per second per worker; even a 10ns invariant open/close cost per step converts the headline 220K states/sec benchmark into ~50K-100K. That violates the brief's "must keep `cargo build --release` working AND not regress perf" gate.

The shadow file (`shard_methods.rs`) sidesteps this by taking `Tracked<&PermissionU64>` as a *function parameter*, expecting the caller to have already done the invariant open. That pattern works inside Verus but would force every caller of `FingerprintShard::contains_or_insert` (the runtime worker loop, the batch APIs at lines 1507-1530) to also be Verus-annotated and to also open the invariant. The blast radius reaches `runtime.rs`, `work_stealing_queues.rs`, and most of the `storage/` module.

#### Gap 3 update (2026-05-13): logically-atomic specs sidestep the per-step open

`vstd::logatom.rs` provides the framework — `ReadOperation` / `MutOperation` / `ReadLinearizer` / `MutLinearizer` traits — for logically-atomic specifications. The pattern (familiar from Iris and CAP) is:

- Define `ContainsOp { fp: u64 }` implementing `ReadOperation` with `Resource = ProtocolState` and `ExecResult = bool`.
- The exec method takes a linearizer as a ghost parameter: `fn contains(&self, fp: u64, Tracked(lin): Tracked<L>) -> bool where L: ReadLinearizer<ContainsOp>`.
- The linearization point is internal — at the CAS site where the contains-or-insert decision becomes externally visible.
- No `open_atomic_invariant!` per probe step. The hot path is exec-rate, the proof obligation is the linearizer's `apply` method which executes only conceptually.

This is the modern alternative to the `arc.rs`-style `open_atomic_invariant!` per CAS pattern. Both work; logatom is preferable for our 10-100M-probe/sec hot path because the abstraction barrier ends at the linearizer call, not at every probe step.

The blast radius into `runtime.rs` shrinks too: callers thread a linearizer down via generic `L` parameters instead of opening the invariant themselves. The linearizer is a single ghost value; the runtime can construct one once at engine startup.

The original gap-3 framing assumed `AtomicInvariant` open-close per probe was the only mechanism. It is not. The choice between the two patterns is a design decision driven by perf-vs-proof-complexity tradeoffs, not a vstd capability gap.

## What a fully-verified shipping code path *would* look like

Concretely: **a Verus `state_machines!` reformulation of the fingerprint store** (Beillahi 2021, AWS s2n-quic verification work, Hance et al. "Storage Systems are Distributed Systems" SOSP 2020). This is the same v1.2.0+ research path called out for T13.5's unbounded-fairness reader liveness in `reader_liveness.rs`.

In a `state_machines!` formulation:
- The whole shard becomes a state machine `FingerprintShardSM` with transitions `insert(fp)`, `begin_resize`, `rehash_one(idx)`, `finalize_resize`, `read(fp)`.
- The pointer-swap and mmap allocation drop out — they're physical realisations of the abstract `begin_resize` and `finalize_resize` transitions.
- Linear ghosts are tied to *transitions*, not to method calls, so the `&self` / linear-token tension dissolves.
- `state_machines!` natively supports temporal invariants and liveness, so the seqlock retry loop's reader liveness and the CAS soundness get proved in the same framework.

Best done jointly with the T13.5 liveness-axiom discharge since both share the same machinery.
